Human-Computer Interaction Project - Laundro-Map

Project group: Julian Tanguma, Leo Moran, Josh Meier, Ezra Barber

Goal:

We would like to solve the issue of chaos in the laundry room, as the laundry room can always be hectic with entire buildings sharing a set amount of washers and dryers. Our project aims to show which laundry machines are available and which are not. It will also show how long until they will be free.



The prototype addresses key user needs by providing location-based navigation, allowing users to select their dorm and view machine availability with clear indicators. It includes a timer and queue management system to streamline access to machines and prevent conflicts. The reporting feature enables users to notify others about broken machines, while the simple and intuitive interface ensures ease of use. These features align with the feedback from interviews, emphasizing convenience, clarity, and personalization.



Human-Computer Interaction Project - Laundro-Map - Prototype 2



Final Three Requirements

(Top) Users can see how much time is left on each machine in their dorm.

Justification: This is a core feature of the website, allowing users to check machine availability remotely. Without it, users would have no way of knowing whether a machine will be free at a certain time, making it harder to plan laundry schedules.

(Middle) Users can see when machines are out of order and/or report a broken machine.

Justification: This feature helps users plan ahead by identifying broken machines, preventing wasted trips to the laundry room.

(Bottom) Users can select a dorm/building preference to display upon login.

Justification: By allowing users to set a default dorm preference, they can quickly access relevant machine availability information.

How the Prototype Addresses Each Requirement

  • (Top Left) Dorm Preference Selection: Users can set their dorm building as a default, ensuring that when they log in, the laundry dashboard displays their preferred location without extra navigation.
  • (Top Right) Out-of-Order Machines: Machines that are broken are clearly labeled in the interface with a distinct visual indicator (e.g., red or grayed-out status). This helps users identify unavailable machines instantly.
  • (Bottom Left) Time Remaining on Machines: The prototype includes a live status tracker for each laundry machine, displaying the remaining time in an intuitive interface. Users can quickly see which machines will be available soon.

Analysis Process

  • Axial Coding: The team identified common themes in user responses, grouping feedback into categories such as availability tracking, ease of use, and error prevention.
  • Affinity Diagramming: A visual mapping of user needs and pain points helped structure key design decisions, ensuring the interface addressed the most pressing usability concerns.

UX Research Process

  • Semi-structured interviews were conducted to understand laundry habits and frustrations.
  • Focus groups were held to refine feature priorities.
  • Users interacted with prototypes to give immediate feedback about app structure and development.

Interaction Flows

  • (Top Left) Setting Dorm Preferences: Users can set their dorm preference for easy future access. (WIP)
  • (Top right) Identifying Out-of-Order Machines: machine indicators that show broken/working machines.
  • (Bottom Left) Checking Machine Availability: Users log in and can slick on their building to see which machines are available.



Results from UX Evaluation

What Worked Well:

  • Overall Task Flow: Users found most tasks “pretty straightforward” and “intuitive,” especially setting a default dorm, which was completed very quickly (about 1 minute as noted in the UX Metrics).
  • Default Setting Feature: Participants appreciated that once they set a default, “it would take you straight into Watson” and eliminate the need to pick a location every time.
  • Real-Time Feedback: The timer and notification features were generally well received, supporting the core function of tracking machine availability.

What Didn’t Work Well:

  • Report Button Issues: Several users were confused by the report button’s design. One participant noted, “there's no back button,” and another mentioned issues with how to select the specific machine for reporting.
  • UI Clarity: The use of gray boxes to group washer and dryer options led to confusion. One interviewee explained, “the fact that they're both grouped in a little gray box” made it hard to distinguish individual elements.
  • Metadata Input: Participants expressed frustration having to manually input machine details. As one said, “I definitely think I was confused… I was just disappointed that I had to pick which dryer was the one that wasn't working for me.”



How the Prototype Improved Based on UX Evaluation

  • UI Adjustments: (Top) The design was refined to remove or rework the confusing gray boxes. For example, feedback like, “maybe getting rid of those gray boxes would make it easier” was directly addressed.
  • Report Function Redesign: (Middle) The report feature was updated by repositioning the report button and adding a clear “back” or cancel option. One participant’s exact feedback—“there's no back button”—prompted this change.
  • Metadata Automation: (Bottom) To ease the reporting process, the prototype was improved so that machine details are now auto-associated, reducing the need for manual entry. This change reflects the feedback where a user noted, “I would prefer metadata” to be automatically captured.

UX Evaluation Tasks

The evaluation included the following tasks:

  • Task 1: Finding a washer/dryer, setting timers, and exploring the interface.
  • Task 2: Reporting a broken machine.
  • Task 3: Setting a default dorm/building preference.

These tasks were chosen to directly test the three core requirements of the prototype (machine timer display, reporting functionality, and default building selection).

UX Evaluation Process

Evaluation Method: A focus group was conducted with two stakeholders who are college students from Watson and Norse Dorm Halls.

Process Details: Participants engaged in a Focus Group session, sharing real-time feedback while completing each task. The session was recorded (with permission), and informal UX metrics were collected via timestamps, observational notes, and tally marks.

Protocol Reference: The session followed the structured guidelines as set out in the UX Evaluation Protocols document.

Future Steps

Based on the evaluation, the following future steps are proposed that go beyond mere prototype refinements:

  • Enhanced Reporting Feature:
    • Functional Improvements: Incorporate an automatic association of machine metadata to eliminate manual input, and add a clear back or cancel button to resolve the “no back button” issue.
  • UI Design Overhaul:
    • Revise the grouping and labeling of washer/dryer options to remove the misleading gray boxes.
    • Add an actual floor mapping of the washrooms.
  • Additional Functionality:
    • Explore integrating more detailed notifications (e.g., real-time queue information) and potential accessibility enhancements.
  • System Requirements Consideration:
    • Some suggestions (e.g., additional accessibility features or data encryption for sensitive inputs) will require backend system changes beyond the Figma prototype improvements.
